# RAG 知识库架构说明

## 一、为什么需要 RAG？

### 大模型的固有局限

大语言模型（LLM）虽然能力强大，但有三个关键问题：

| 问题 | 说明 | 影响 |
|---|---|---|
| **知识截止** | 模型训练数据只到某个时间点 | 不知道最新业务策略、产品更新 |
| **缺乏私域知识** | 模型没看过你的企业文档 | 不知道你的 SOP、定价策略、客户案例 |
| **幻觉** | 模型会编造看起来合理但实际错误的内容 | 产生不符合公司政策的建议 |

### RAG 的解决思路

**RAG（Retrieval-Augmented Generation）** 的核心思想是：**不依赖模型记住你的知识，而是让模型在回答时「查资料」**。

```
传统 LLM 调用：
用户问题 → LLM → 回答（全靠模型记忆）

RAG 调用：
用户问题 → 检索知识库 → 找到相关片段 → 问题 + 片段 → LLM → 回答
```

在我们的场景中：
- 客户风险 Agent 要生成经营建议 → 先查知识库中对应的策略文档
- 低登录客户的建议 → 检索 `low_login_solutions.md` 中的具体做法
- Agent 的回答就有了企业知识支撑，不再凭空编造

## 二、检索流程详解

### 完整流程图

```
┌─────────────────────────────────────────────────────────────┐
│                     知识库文档（.md）                        │
│  customer_retention.md, renewal_strategy.md, ...            │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│              文档切块（RecursiveCharacterTextSplitter）      │
│  每段 600 字符，重叠 120 字符                              │
│  按标题层级（## → ### → 段落 → 句子）切分                  │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│              Embedding（向量化）                             │
│  每个文本块 → 384/768 维向量                                │
│  "低登录率解决..." → [0.23, -0.56, 0.89, ...]              │
└────────────────────┬────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│              FAISS 向量索引                                  │
│  存储所有向量，支持毫秒级相似度搜索                         │
│  保存到 storage/faiss_index/ 目录                           │
└────────────────────┬────────────────────────────────────────┘
                     ↑
              ┌──────┴──────┐
              │  检索阶段    │
              └──────┬──────┘
                     ↑
┌─────────────────────────────────────────────────────────────┐
│  用户查询 → Embedding（同一模型）→ 向量 → FAISS 相似度搜索  │
│  "客户登录天数低" → [0.31, -0.42, ...] → 找到最近似的 2 个块   │
└─────────────────────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────┐
│  检索结果 → 注入 Prompt → LLM → 有知识支撑的输出            │
└─────────────────────────────────────────────────────────────┘
```

### 代码层面的调用流程

```python

# 3. 将 context 注入 LLM 的 Prompt
messages = [
    {"role": "system", "content": "你是客户成功经理..."},
    {"role": "user", "content": f"客户数据：...\n\n参考知识库：\n{context}\n\n请分析..."}
]
```

## 三、Embedding 是什么？

### 通俗解释

**Embedding 就是把文字变成向量（数字数组）**。

- 「猫」→ [0.1, 0.5, -0.2, ...]
- 「狗」→ [0.2, 0.4, -0.1, ...]
- 「汽车」→ [0.8, -0.3, 0.6, ...]

相似的文本在向量空间中距离更近。「猫」和「狗」的距离 < 「猫」和「汽车」的距离。

### 技术要点

- **维度**：本项目使用 `paraphrase-multilingual-MiniLM-L12-v2`，输出 384 维向量
- **多语言支持**：该模型支持中文、英文等 50+ 语言，中文查询可以匹配中文文档
- **对称 vs 非对称检索**：我们用的是对称检索（查询和文档是同一语言、相似风格），效果已经足够

### 为什么不用 DeepSeek/OpenAI 的 Embedding API？

| 方案 | 优点 | 缺点 |
|---|---|---|
| 本地模型（本项目的方案） | 免费、离线、无数据泄漏 | 首次需要下载模型（约 470MB） |
| API Embedding | 无需下载、效果更稳定 | 额外付费、依赖网络、数据传输 |

在企业本地部署场景下，**本地模型 + 私有知识库** 是标准实践。

## 四、FAISS 是什么？

### 一句话

**FAISS 是一个向量搜索引擎，能在百万级向量中快速找到「最相似」的那些。**

全称：Facebook AI Similarity Search（Meta AI 开源）

### 核心原理

传统数据库用 B-Tree 按值搜索（找 `age = 25`），但无法做「找最相似向量」。

FAISS 通过两种技术加速：

1. **IVF（倒排索引）**：把向量空间划分成多个区域（类似图书分类），搜索时只看相关的几个区域
2. **量化压缩**：压缩向量精度，牺牲少量准确率换取数十倍的搜索速度

### 在本项目中的角色

```
知识库 20 个文档 → 切块 → 约 60-80 个文本块 → 向量化 → FAISS 索引
                                                      ↓
用户查询 → 向量化 → FAISS 搜索（1-5 毫秒）→ 返回 Top-K 最相关文本块
```

### 持久化

FAISS 索引保存到 `storage/faiss_index/` 目录：

```
storage/faiss_index/
├── index.faiss     # 向量数据
└── index.pkl       # 文档原文映射
```

- 首次运行：自动构建索引（下载模型 + 向量化，约 1-3 分钟）
- 之后启动：直接加载已保存的索引（秒级）

## 五、企业里如何使用 RAG

### 典型架构

```
                  ┌─────────────┐
                  │  用户/Agent  │
                  └──────┬──────┘
                         │ 提问
                         ▼
               ┌─────────────────┐
               │   RAG Service    │
               │   (langchain)   │
               └──┬──────────┬───┘
                  │          │
          ┌───────▼──┐  ┌───▼────────┐
          │ 知识库     │  │  大模型    │
          │ (FAISS)   │  │ (DeepSeek) │
          └───────────┘  └────────────┘
```

### 知识库的生命周期

1. **构建阶段**：业务专家撰写知识文档 → CSM 负责人审核 → 上传到 `knowledge_base/`
2. **运行阶段**：RAG Service 自动加载 → 切块向量化 → 存入 FAISS
3. **更新阶段**：文档内容变更后，调用 `rag.rebuild_index()` 重建索引
4. **版本管理**：知识库文档应当纳入 Git 管理，与代码一起版本化

### 最佳实践

| 实践 | 说明 |
|---|---|
| **文档粒度** | 每篇文档聚焦一个主题，便于精确检索 |
| **标题规范** | 使用 `##` 分级标题，切块策略会按标题保留语义边界 |
| **定期更新** | 业务策略变化后及时更新知识库，并重建索引 |
| **质量把控** | 知识库内容需经过评审，错误的知识会误导 Agent |
| **效果监控** | 定期检查 Agent 的输出是否准确引用了知识库内容 |

### 进阶方向

- **Hybrid Search**：结合关键词搜索（BM25）和向量搜索，提高精确率
- **Rerank**：对检索结果重排序，把最相关的排到最前面
- **Agentic RAG**：Agent 可以自己决定搜什么、搜几次、要不要追问
- **Multi-模态 RAG**：支持图片、表格等非文本内容的检索

## 六、本项目实现总结

| 组件 | 技术选型 | 说明 |
|---|---|---|
| 知识库 | Markdown 文档 | 4 篇 CS 策略文档 |
| 文档加载 | LangChain DirectoryLoader | 自动扫描目录 |
| 文本切块 | RecursiveCharacterTextSplitter | 600 字符/块，120 重叠 |
| Embedding | sentence-transformers | 多语言模型，384 维 |
| 向量存储 | FAISS | 本地索引，毫秒级检索 |
| 服务封装 | RAGService | query() / build_context() / diagnose_and_retrieve() |
| Agent 集成 | CustomerRiskAgent | 分析前自动检索知识库 |
